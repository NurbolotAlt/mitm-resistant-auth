# SecureNotes — MITM-Resistant Authentication (Proof of Concept)

A full-stack notes application built to demonstrate an **out-of-band key exchange** scheme that
resists man-in-the-middle attacks on an unencrypted channel.

The notes app itself is deliberately simple. The interesting part is the **authentication
protocol**: a user's password is never transmitted in a form that a network eavesdropper can
read, and the shared secret used to protect it is never fully transmitted over the network at
all — half of it travels over a completely separate channel (email).

> ⚠️ **Academic proof of concept.** This project was built to explore and demonstrate a
> security concept. It is not hardened for production use — see
> [Known limitations](#known-limitations).

---

## The problem

In a classic Diffie–Hellman exchange over a plain HTTP connection, an attacker positioned
between the client and the server can intercept both halves of the exchange, negotiate separate
keys with each party, and transparently relay traffic — reading and modifying everything in
between. The exchange is only as trustworthy as the channel it happens on.

## The approach

This project splits the key exchange across **two independent channels**, so that compromising
the network alone is not enough to recover the shared secret.

The shared key is assembled from two halves:

| Half | Generated by | Travels over |
|---|---|---|
| `serverNonce` | Server | HTTP |
| `clientNonce` | Client | **Email (out-of-band)** |

```
shared key = serverNonce + clientNonce
```

An attacker who fully controls the HTTP connection observes `serverNonce` but never sees
`clientNonce`, because that half is delivered by the user through their own email account to a
mailbox the server reads via the Gmail API. Without both halves, the intercepted password
ciphertext cannot be decrypted.

### Registration flow

```
 Client                                  Server                      Gmail
   │                                        │                          │
   │ ── POST /reg { username } ───────────► │                          │
   │                                        │  generate serverNonce    │
   │ ◄──────────────── { serverNonce } ──── │                          │
   │                                        │                          │
   │  generate clientNonce                  │                          │
   │  key = serverNonce + clientNonce       │                          │
   │                                        │                          │
   │ ══ email: subject=username ═══════════════════════════════════►   │
   │           body=clientNonce             │        (out-of-band)     │
   │                                        │                          │
   │ ── POST /checkEmail { username } ────► │ ── read unread thread ─► │
   │                                        │ ◄─ clientNonce ───────── │
   │                                        │  key = serverNonce +     │
   │ ◄──────────────────────── { found } ── │        clientNonce       │
   │                                        │                          │
   │  ciphertext = encrypt(password, key)   │                          │
   │ ── POST /sendPassword { ciphertext } ► │  decrypt with key,       │
   │                                        │  create user             │
   │ ◄─────────────────────────────── OK ── │                          │
```

Login follows the same pattern against `/auth`, `/checkEmailLog` and `/sendPasswordLog`.
Once a session key is established, note contents are also enciphered with it in both
directions (`/SubmitData`, `/ReceiveData`).

## Tech stack

**Frontend** — React 18 (Create React App), React Router, Axios, React Icons
**Backend** — Node.js, Express, Mongoose
**Database** — MongoDB
**External** — Gmail API via `googleapis` (OAuth2), Nodemailer

├── src/                              # React frontend

## API

| Method | Endpoint | Purpose |
|---|---|---|
| `POST` | `/reg` | Begin registration, return `serverNonce` |
| `POST` | `/auth` | Begin login, return `serverNonce` |
| `POST` | `/checkEmail` | Poll Gmail for the out-of-band `clientNonce` (registration) |
| `POST` | `/checkEmailLog` | Poll Gmail for the out-of-band `clientNonce` (login) |
| `POST` | `/sendPassword` | Submit enciphered password, create account |
| `POST` | `/sendPasswordLog` | Submit enciphered password, verify credentials |
| `POST` | `/SubmitData` | Save enciphered note content |
| `POST` | `/ReceiveData` | Retrieve enciphered note content |

## Getting started

### Prerequisites

- Node.js 16+
- A MongoDB instance (local or Atlas)
- A Google Cloud project with the **Gmail API** enabled and OAuth2 credentials

### Setup

```bash
git clone https://github.com/NurbolotAlt/notesapp.git
cd notesapp

# Frontend
npm install

# Backend
cd Backend
npm install
```

Create `Backend/.env`:

```env
DB_URI=your_mongodb_connection_string
PORT=4000
CLIENT_ID=your_google_oauth_client_id
CLIENT_SECRET=your_google_oauth_client_secret
REDIRECT_URI=https://developers.google.com/oauthplayground
REFRESH_TOKEN=your_google_oauth_refresh_token
```

Set the mailbox the server reads from in `Backend/constants.js`.

### Run

```bash
# Terminal 1 — backend on :4000
cd Backend && node index.js

# Terminal 2 — frontend on :3000
npm start
```

### Using it

1. Register with a username and password.
2. On the verification screen, send an email to the configured mailbox with the **username as
   the subject** and the displayed **code as the body**.
3. Press **Submit**. The server reads the code from the mailbox, completes the key exchange,
   and creates your account.
4. Log in and use the notes editor.

## Known limitations

This is a concept demonstration, and the following are deliberately out of scope or known
weaknesses:

- **The cipher is a repeating-key XOR**, not a real cryptographic primitive. It illustrates
  the key-exchange idea but offers little real confidentiality. A production version would use
  AES-GCM or similar with a properly derived key (HKDF).
- **Passwords are stored in plaintext** in MongoDB. They should be hashed with bcrypt or Argon2.
- **Server state is global**, not per-session — the current implementation handles one user
  at a time and would race under concurrent use. Sessions belong in a store keyed by session ID.
- **No transport encryption.** The whole point is to explore what can be achieved *without*
  TLS, but any real deployment should use HTTPS regardless.
- **Email is a slow, manual channel.** It proves the out-of-band concept; a practical system
  would use a TOTP app, push notification, or QR code for the same purpose.
- Nonces are 6 characters from a 62-character alphabet — fine for a demo, too small for real use.
- No rate limiting, input validation, or CSRF protection.

## Roadmap

- [ ] Replace XOR with AES-GCM and derive keys via HKDF

- [ ] Per-session server state and proper session management
- [ ] Swap the email channel for a TOTP/push-based second channel
